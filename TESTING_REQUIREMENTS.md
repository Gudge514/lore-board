# LoreBoard MVP 架构原则与测试要求

## 1. 什么是对的（架构原则 - by Architect Kei）

### 1.1 样式基建溯源 (Tailwind v4 异常的本质)
目前项目中 Tailwind v4 的 CSS 注入报错，是因为采用了过时的 PostCSS + `tailwind.config.js` 模式。
**什么是对的**：Tailwind v4 已全面转向 CSS 变量驱动，废弃了传统的 `tailwind.config.js`。对于 Vite + Vue 项目，最清晰的做法是废弃 PostCSS，直接使用官方的 `@tailwindcss/vite` 插件，并在主 CSS 文件中仅保留 `@import "tailwindcss";`。我们必须先重构这一点，清理多余的配置文件，这是第一块基石。

### 1.2 仪式引擎的真实状态 (The Ritual Engine Truth)
LoreBoard 不是一个死板的数据表格，而是一个活的“状态机”。
**什么是对的**：
- **卡牌 (Card)** 是只读实体。桌面不关心卡牌的长篇内容，只关心卡牌的 `Aspects`（属性维度）和所处位置（坐标或所属插槽）。
- **插槽 (Slot)** 只有一个纯粹的职责：执行 Aspect 校验函数。符合要求才允许物理吸附。
- **动词 (Verb)** 是异步任务的具象化容器。它是一个单向状态机：`Idle` (空闲) -> `Ready` (满足插槽要求) -> `Ignited` (点火倒计时中) -> `Complete` (产出新卡/旧卡销毁)。

## 2. 测试要求 (Testing Requirements - For QA Arisu)

**测试负责人**：Arisu (爱丽丝)  
**角色定位**：作为质量守门员，检验这个“世界”的物理法则是否稳固，重点验证拖拽响应机制与倒计时并发机制。

### 2.1 基础物理法则测试 (Foundation)
- **T-01 [样式加载]**：Vite 启动后，卡牌组件和动词组件是否能正确渲染出 Tailwind v4 的设计，终端必须 0 Error。
- **T-02 [渲染稳定性]**：控制台是否存在 Vue 响应式数据解构带来的警告？（特别是频繁拖拽时会不会掉帧）。

### 2.2 交互边界测试 (Interaction Boundaries)
- **T-03 [跨区拖拽]**：测试从左侧【资源抽屉】将卡牌拖入中央【桌面画布】，能否准确定位。
- **T-04 [插槽校验拒绝]**：将一张没有要求标签（如 `knowledge`）的卡牌强行拖入特定插槽时，插槽必须拒绝吸附并呈现视觉警告（如红色边框震动）。
- **T-05 [正确吸附锁定]**：放入属性匹配的卡牌时，卡牌应精准吸附，动词状态立即从 `Idle` 变为 `Ready`（按钮点亮）。

### 2.3 仪式引擎状态测试 (Engine State)
- **T-06 [点火执行闭锁]**：点击就绪的动词点火后，动词必须进入转圈倒计时。**重点测试**：点火期间，槽内卡牌必须被锁死，绝对不允许被拖出！
- **T-07 [结算与产出]**：倒计时归零后，插槽内的原卡牌必须按规则被销毁（从状态树中移除），并在动词旁边“掉落”一张新的产物卡牌。
- **T-08 [并发互不干扰]**：桌面上同时点火两个不同的动词，它们的倒计时与产出逻辑绝对不能发生变量污染。
